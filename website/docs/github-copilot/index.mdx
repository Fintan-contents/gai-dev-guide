## 検討事項など

- 読み手は以下
  - PM：予算含めた、GHC導入の決定者
  - アーキ：GHC導入の技術観点のアドバイスをPMにする人、導入決定後の開発プロセス検討・メンバー説明を担う人
  - アプリチームのリーダー：アーキから説明を受けてメンバーにGHC使い方を教える人
  - アプリチームのメンバー：アーキ・アプリチームのリーダーから説明を受けて、実際にGHCを使う人
- ユースケースは以下 ※括弧はユースケースの主語、★がメイン
  - A）PJにてGHC導入を検討する（★PM、アーキ）
  - B）PJにてGHC利用環境を整備する（★アーキ）
    - B-1）利用者としてGHCの使い方を理解し、使えるようにする（★アーキ）
    - B-2）PJ個別理由を考慮して、PJ個別に最適化されたガイドを作成する（★アーキ）
    - B-3）PJ個別最適化されたガイドをメンバーに説明する（★アーキ、アプリチームのリーダー）
  - C）GHCの使い方を理解し、最低限使えるようにする（アーキ、★アプリチームのリーダー、アプリチームのメンバー）
  - D）GHCを使いこなす（アーキ、アプリチームのリーダー、★アプリチームのメンバー）
- 機能紹介はペルソナで分離が良い
  - Copilotの機能のうち、管理者(アーキ)向けか、利用者(アプリチーム)は別れていたほうが良い（役割をまたいでガイドを読む人は少数だから）
  - [GitHub Copilot の機能 | GitHub Docs](https://docs.github.com/ja/copilot/about-github-copilot/github-copilot-features)を見て思った
- 設定はペルソナ分離がいいのか検討
  - もしDPCでお膳立てしていて、アーキでの組織設定がほぼ不要なら、分離しなくて良い
    - ライセンス関連のサポートについて
- 追記したい、[GitHub Copilot コード レビューの使用 | GitHub Docs](https://docs.github.com/ja/copilot/using-github-copilot/code-review/using-copilot-code-review)を
  - [Configuring coding guidelines for GitHub Copilot code review | GitHub Docs](https://docs.github.com/ja/copilot/using-github-copilot/code-review/configuring-coding-guidelines)
